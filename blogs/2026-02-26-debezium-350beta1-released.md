<div class="paragraph"> <p>It&#8217;s been a few weeks since our first pre-release for Debezium 3.5, and we&#8217;re pleased to announce that the next installment is now available: <strong>3.5.0.Beta1</strong>. With an extended cadence since the first alpha, this release is packed with dozens of new features, improvements, and bug fixes. Let&#8217;s dive into what&#8217;s new!</p> </div> <div class="paragraph"> <p><!--- more --></p> </div> <div class="sect1"> <h2 id="new-features-and-improvements">New features and improvements</h2> <div class="sectionbody"> <div class="sect2"> <h3 id="debezium_core">Debezium Core</h3> <div class="sect3"> <h4 id="new_parallel_multithreaded_chunked_based_snapshots">New parallel, multithreaded chunked-based snapshots</h4> <div class="paragraph"> <p>A few years ago, Debezium introduced its first iteration of parallel snapshotting. We&#8217;re happy to report that the next evolution of this feature is here. The new parallel snapshotting implementation takes the existing solution a step further, allowing you to snapshot a single table across multiple threads using chunking (<a href="https://github.com/debezium/dbz/issues/1220">debezium/dbz#1220</a>).</p> </div> <div class="paragraph"> <p>This feature provides several key guarantees:</p> </div> <div class="ulist"> <ul> <li> <p>All snapshot threads are fully utilized for the entire duration of the snapshot, maximizing resource utilization.</p> </li> <li> <p>All tables, regardless of row count, can take advantage of parallel snapshotting.</p> </li> </ul> </div> <div class="paragraph"> <p>The feature is controlled by the following configuration options:</p> </div> <div class="dlist"> <dl> <dt class="hdlist1"><code>snapshot.max.threads</code></dt> <dd> <p>The number of threads used to parallelize the snapshot. Defaults to <code>1</code>.</p> </dd> <dt class="hdlist1"><code>snapshot.max.threads.multiplier</code></dt> <dd> <p>A global multiplier that determines how many chunks to create for each table. When set to <code>1</code> (the default), Debezium computes <em>(row count / snapshot max threads)</em> to determine the number of chunks. For example, a table with 1M rows and 4 snapshot threads produces 4 chunks of 250,000 rows each.<br /> <br /> In scenarios where you cannot increase the number of snapshot threads but need more granular chunking, increase this multiplier. Using the same 1M-row table with 4 threads and a multiplier of 2, Debezium creates 8 chunks of 125,000 rows each.</p> </dd> <dt class="hdlist1"><code>snapshot.max.threads.multiplier.&lt;fully_qualified_table_name&gt;</code></dt> <dd> <p>A table-specific multiplier that overrides the global multiplier. This allows individual tables to be scaled differently, using more or fewer chunks to accommodate varying table sizes or resource constraints.</p> </dd> </dl> </div> <div class="admonitionblock tip"> <table> <tr> <td class="icon"> <i class="fa icon-tip" title="Tip"></i> </td> <td class="content"> <div class="paragraph"> <p>While both the thread count and multiplier can be tuned, increasing the number of threads is generally the most effective way to improve throughput.</p> </div> </td> </tr> </table> </div> <div class="paragraph"> <p>You can also use chunk-based snapshotting without parallelism. Simply configure one or both <code>snapshot.max.threads.multiplier</code> variants to control the chunk count while keeping a single thread.</p> </div> <div class="paragraph"> <p>For existing users who have not set <code>snapshot.max.threads</code> or <code>snapshot.max.threads.multiplier</code>, connectors will continue to operate exactly as before, using the sequential single-threaded snapshot mechanism. No configuration changes are needed.</p> </div> <div class="paragraph"> <p>For users who previously set <code>snapshot.max.threads</code> and prefer the older table-per-thread behavior, set <code>internal.legacy.snapshot.max.threads</code> to <code>true</code>. The legacy parallel snapshot feature will remain available for some time but will eventually be removed in favor of the chunk-based approach.</p> </div> <div class="admonitionblock important"> <table> <tr> <td class="icon"> <i class="fa icon-important" title="Important"></i> </td> <td class="content"> <div class="paragraph"> <p>The chunk-based parallel snapshot feature has the following limitations:</p> </div> <div class="ulist"> <ul> <li> <p>Tables that use snapshot select overrides</p> </li> <li> <p>Tables without a primary key</p> </li> </ul> </div> <div class="paragraph"> <p>In these cases, the algorithm assigns a single chunk to these tables, effectively falling back to the table-per-thread strategy.</p> </div> </td> </tr> </table> </div> </div> </div> <div class="sect2"> <h3 id="debezium_for_mysql">Debezium for MySQL</h3> <div class="sect3"> <h4 id="modify_binlog_offset_position_using_signals">Modify binlog offset position using signals</h4> <div class="paragraph"> <p>One of the community&#8217;s most requested features is the ability to adjust a connector&#8217;s offset position without directly manipulating the offset topic. The <code>&lt;data&gt;</code> payload varies depending on whether the MySQL source database uses GTIDs.</p> </div> <div class="listingblock"> <div class="title">Inserting a SQL source signal to change the binlog position</div> <div class="content"> <pre class="CodeRay highlight"><code><span class="class">INSERT</span> <span class="class">INTO</span> signal_table <span class="keyword">values</span> class="delimiter">&quot;</span></span>: <span class="integer">1234</span> }</code></pre> </div> </div> <div class="paragraph"> <p>Based on community feedback, we plan to roll this feature out to additional connectors in future releases.</p> </div> </div> </div> <div class="sect2"> <h3 id="debezium_for_oracle">Debezium for Oracle</h3> <div class="sect3"> <h4 id="oracle_memory_improvements">Oracle memory improvements</h4> <div class="paragraph"> <p>The Debezium Oracle connector buffers transaction events due to how its CDC integration with Oracle works. This buffer stores column values for each event along with various bookkeeping metadata used for populating the event&#8217;s <code>source</code> information block and managing the buffer lifecycle.</p> </div> <div class="paragraph"> <p>While the buffer&#8217;s footprint is primarily driven by event column values, we&#8217;ve identified several ways to reduce the bookkeeping overhead — either unconditionally or optionally via configuration (<a href="https://github.com/debezium/dbz/issues/1425">debezium/dbz#1425</a>).</p> </div> <div class="paragraph"> <p>No configuration changes are required for existing users. You should see approximately a <strong>50% reduction</strong> in the buffer&#8217;s bookkeeping footprint, which translates to a lower overall memory footprint for your connector deployment. The improvement is especially noticeable in environments with large transactions.</p> </div> <div class="admonitionblock tip"> <table> <tr> <td class="icon"> <i class="fa icon-tip" title="Tip"></i> </td> <td class="content"> <div class="paragraph"> <p>An additional 8% bookkeeping reduction can be achieved by setting <code>log.mining.buffer.track.rs_id</code> to <code>false</code>. This is only recommended if you do not need <code>source.rs_id</code> populated in your change events.</p> </div> </td> </tr> </table> </div> <div class="paragraph"> <p>For a deeper look at the impact of these changes, see the <a href="https://github.com/debezium/debezium/pull/7065">pull request</a> description.</p> </div> </div> <div class="sect3"> <h4 id="new_logminer_advancement_strategy">New LogMiner advancement strategy</h4> <div class="paragraph"> <p>Some Oracle environments inevitably have long-running or large transactions that cannot be avoided. Due to LogMiner&#8217;s requirements, Debezium interacts with it in a way that guarantees transaction and data consistency — but this can delay low watermark advancement.</p> </div> <div class="paragraph"> <p>The community has requested more control over when Debezium advances the low watermark, even while long-running or large transactions are in flight. Debezium 3.5 introduces a time-based threshold for controlling low watermark advancement (<a href="https://github.com/debezium/dbz/issues/1553">debezium/dbz#1553</a>).</p> </div> <div class="paragraph"> <p>This feature works similarly to transaction retention: rather than discarding the transaction, it is retained, but the low watermark is advanced. Enable it by setting <code>log.mining.window.max.ms</code> to any value greater than zero.</p> </div> <div class="admonitionblock warning"> <table> <tr> <td class="icon"> <i class="fa icon-warning" title="Warning"></i> </td> <td class="content"> <div class="paragraph"> <p>LogMiner has specific requirements to guarantee data consistency. Advancing the SCN with this feature introduces risks because those requirements are no longer fully honored. This can lead to transactions that fail to commit, events that appear as unsupported operations, or other LogMiner anomalies.</p> </div> <div class="paragraph"> <p>Using this feature acknowledges the above risks. Please carefully evaluate the trade-offs before enabling it in any production pipeline.</p> </div> </td> </tr> </table> </div> </div> <div class="sect3"> <h4 id="incremental_snapshot_performance_improvements">Incremental snapshot performance improvements</h4> <div class="paragraph"> <p>Debezium&#8217;s incremental snapshot process relies heavily on primary key indexes for efficient row lookup between a chunk&#8217;s lower and upper boundaries. As table sizes grow, this efficiency becomes increasingly critical.</p> </div> <div class="paragraph"> <p>A common source of performance degradation with incremental snapshots involves composite keys on very large tables. The SQL predicates required to locate rows between boundaries with composite keys are complex, and not every database engine can optimize them to efficiently use the primary key index.</p> </div> <div class="paragraph"> <p>For Oracle, one way to improve performance is to use <code>ROWID</code> — a unique byte offset for each row — instead of the primary key index. This bypasses the index entirely and operates at the lowest level of the database to isolate the target chunk of rows during incremental snapshots (<a href="https://github.com/debezium/dbz/issues/1108">debezium/dbz#1108</a>).</p> </div> </div> </div> <div class="sect2"> <h3 id="debezium_for_sql_server">Debezium for SQL Server</h3> <div class="sect3"> <h4 id="managed_identity_support">Managed identity support</h4> <div class="paragraph"> <p>The SQL Server connector now supports identity-based authentication via the <code>driver.authentication</code> configuration property (<a href="https://github.com/debezium/dbz/issues/1153">debezium/dbz#1153</a>). Set this property to <code>ActiveDirectoryManagedIdentity</code> or the legacy <code>ActiveDirectoryMSI</code> to connect to Azure SQL databases.</p> </div> </div> <div class="sect3"> <h4 id="improved_transaction_metadata_handling">Improved transaction metadata handling</h4> <div class="paragraph"> <p>Debezium has historically lacked a reliable way to identify when SQL Server transactions end. As a result, the transaction <code>END</code> marker was emitted only after the first event of the next transaction, leading to delayed delivery in low-traffic systems when using <code>provide.transaction.metadata</code>.</p> </div> <div class="paragraph"> <p>Debezium 3.5 addresses this by leveraging <code>dm_cdc_log_scan_sessions</code> to detect when transactions have completed (<a href="https://github.com/debezium/dbz/issues/1604">debezium/dbz#1604</a>). This allows <code>END</code> markers to be emitted more promptly, particularly in low-traffic environments.</p> </div> <div class="admonitionblock note"> <table> <tr> <td class="icon"> <i class="fa icon-note" title="Note"></i> </td> <td class="content"> <div class="paragraph"> <p>This improvement only applies to connectors deployed with <code>provide.transaction.metadata</code> enabled. It is fully managed by Debezium — no additional configuration changes are needed.</p> </div> </td> </tr> </table> </div> </div> </div> <div class="sect2"> <h3 id="debezium_jdbc_sink">Debezium JDBC sink</h3> <div class="sect3"> <h4 id="postgresql_unnest_support">PostgreSQL unnest support</h4> <div class="paragraph"> <p>PostgreSQL&#8217;s <code>UNNEST</code> function expands an array into a set of rows, and the Debezium JDBC sink connector can now leverage it. Enable this for <code>insert</code> and <code>upsert</code> modes by setting <code>dialect.postgres.unnest.insert.enabled</code> to <code>true</code> (<a href="https://github.com/debezium/dbz/issues/1525">debezium/dbz#1525</a>).</p> </div> <div class="paragraph"> <p>The unnest function allows the JDBC sink connector to write multiple rows in a single JDBC SQL operation, reducing overhead and significantly increasing write throughput.</p> </div> <div class="paragraph"> <p>There are several limitations to be aware of:</p> </div> <div class="ulist"> <ul> <li> <p>Update and delete operations continue to use the standard JDBC batch mechanism.</p> </li> <li> <p>PostgreSQL imposes an upper limit of 65,535 parameters per statement. For tables with many columns, be mindful of the batch size to avoid exceeding this limit.</p> </li> <li> <p>Batches containing a single row fall back to the standard <code>INSERT INTO &#8230;&#8203; ON CONFLICT</code> syntax, as unnest provides no benefit for single-row operations.</p> </li> <li> <p>Because unnest requires all rows in a batch to share the same schema, any change in a table&#8217;s column count forces an early flush of the buffer.</p> </li> </ul> </div> <div class="paragraph"> <p>For most environments, these limitations won&#8217;t be encountered, but they&#8217;re worth keeping in mind.</p> </div> </div> <div class="sect3"> <h4 id="ibmi_support">IBMi support</h4> <div class="paragraph"> <p>As Debezium&#8217;s source connector portfolio continues to grow, our goal is to provide comprehensive parity with the JDBC sink to enable full homogeneous replication between the same source and target databases. With that in mind, the Debezium JDBC sink connector now supports IBMi target databases (<a href="https://github.com/debezium/dbz/issues/1497">debezium/dbz#1497</a>).</p> </div> </div> </div> <div class="sect2"> <h3 id="debezium_server">Debezium Server</h3> <div class="sect3"> <h4 id="nats_jetstream_configurable_naming">NATS JetStream configurable naming</h4> <div class="paragraph"> <p>The Debezium Server NATS JetStream stream name can now be configured through the <code>debezium.sink.nats-jetstream.stream-name</code> deployment property. This property is optional and remains backward compatible — existing deployments continue to use the default <code>DebeziumStream</code> (<a href="https://github.com/debezium/dbz/issues/1581">debezium/dbz#1581</a>).</p> </div> <div class="paragraph"> <p>This change enables multiple Debezium Server instances to share the same NATS JetStream.</p> </div> </div> </div> <div class="sect2"> <h3 id="debezium_platform">Debezium Platform</h3> <div class="sect3"> <h4 id="first_time_guided_tour">First-time guided tour</h4> <div class="paragraph"> <p>When exploring new software for the first time, a guided tour can be a great way to quickly become familiar with what the solution offers.</p> </div> <div class="paragraph"> <p>Debezium 3.5 continues to invest in the user experience with a new guided walkthrough feature in the Platform UI (<a href="https://github.com/debezium/dbz/issues/1244">debezium/dbz#1244</a>). The tour offers two modes: a quick guide for a rapid overview, or an advanced tour that walks you through creating a new pipeline step by step.</p> </div> <div class="videoblock text-center"> <div class="content"> <video controls="controls" src="https://github.com/user-attachments/assets/5ef49099-e52b-401b-a407-9cc17063b13d" width="100%"> Your browser does not support the video tag. </video> </div> </div> </div> </div> <div class="sect2"> <h3 id="debezium_quarkus_extensions">Debezium Quarkus Extensions</h3> <div class="sect3"> <h4 id="support_for_oracle">Support for Oracle</h4> <div class="paragraph"> <p>The Debezium Quarkus extensions now include support for the Oracle connector (<a href="https://github.com/debezium/dbz/issues/1389">debezium/dbz#1389</a>). This allows users to snapshot or backfill historical data from the source database, just like other relational and non-relational connectors. The feature builds on the <code>initial_scan</code> behavior provided by CockroachDB&#8217;s change stream technology.</p> </div> </div> <div class="sect3"> <h4 id="multi_table_concurrent_streaming">Multi-table concurrent streaming</h4> <div class="paragraph"> <p>In earlier builds, the CockroachDB connector created one change stream per table. While functional, CockroachDB recommends keeping the number of changefeed jobs below 80 per cluster — a limit that can be quickly reached in environments with many tables.</p> </div> <div class="paragraph"> <p>Debezium 3.5 introduces the ability to capture changes from multiple tables using a single changefeed job (<a href="https://github.com/debezium/dbz/issues/1628">debezium/dbz#1628</a>). This aligns with CockroachDB&#8217;s recommended thresholds and improves overall throughput of the event capture process.</p> </div> </div> </div> <div class="sect2"> <h3 id="debezium_for_informix">Debezium for Informix</h3> <div class="sect3"> <h4 id="omit_empty_transaction_metadata_events">Omit empty transaction metadata events</h4> <div class="paragraph"> <p>When <code>provide.transaction.metadata</code> is enabled, most consumers are only interested in transactions that contain events matching the connector&#8217;s include list.</p> </div> <div class="paragraph"> <p>Debezium 3.5 introduces <code>cdc.return.empty.transactions</code>, a configuration option that controls whether Informix returns empty transactions (<a href="https://github.com/debezium/dbz/issues/1587">debezium/dbz#1587</a>). By default, empty transactions are omitted. To restore the previous behavior, set this option to <code>true</code>.</p> </div> </div> </div> <div class="sect2"> <h3 id="debezium_for_vitess">Debezium for Vitess</h3> <div class="sect3"> <h4 id="improved_transaction_ordering_with_epoch">Improved transaction ordering with epoch</h4> <div class="paragraph"> <p>The Vitess connector can provide <a href="https://debezium.io/documentation/reference/stable/connectors/vitess.html#vitess-ordered-transaction-metadata">transaction order metadata</a> to help downstream consumers process change events in the correct order. This is particularly important during operations like repartitioning.</p> </div> <div class="paragraph"> <p>In these cases, the epoch value needs to be incremented to ensure downstream consumers rank and order events correctly. Debezium 3.5 introduces <code>vitess.connector.generation</code>, a configuration property that can be increased when making changes that impact ordering semantics. Updating this value triggers an automatic epoch update on connector restart.</p> </div> </div> </div> </div> </div> <div class="sect1"> <h2 id="other-changes">Other changes</h2> <div
